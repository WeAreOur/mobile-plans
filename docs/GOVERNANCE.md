# Governance

## Overview

Mobile Plans is governed by a community-first model that balances openness with quality control.

## Roles and Responsibilities

### Contributors
- **Anyone** can contribute data or improvements
- Must follow contribution guidelines
- Retain attribution for their work
- No special privileges required

### Trusted Contributors
- **Earned role** through consistent, quality contributions
- Can review and approve pull requests
- Help with dispute resolution
- Mentor new contributors

**How to become a Trusted Contributor:**
- 10+ accepted contributions
- History of accurate, well-sourced data
- Positive community interactions
- Nominated by existing Trusted Contributors or Maintainers

### Maintainers
- **Core team** responsible for project direction
- Merge pull requests
- Resolve disputes
- Set and enforce policies
- Manage releases

**Current Maintainers:**
- github:gatsby (founder)

**How to become a Maintainer:**
- Long-term Trusted Contributor
- Deep understanding of project principles
- Community nomination and consensus
- Commitment to charter principles

### Verified Providers
- **Company representatives** with verified accounts
- Can officially maintain their own data
- Subject to same rules as community
- Cannot override community consensus

## Decision-Making Process

### Data Changes
1. Anyone submits a PR
2. Automated validation checks
3. Community review (optional)
4. Trusted Contributor or Maintainer approves
5. Merged

**Timeline:** Usually 24-48 hours

### Policy Changes
1. Proposal opened as GitHub Discussion
2. Community feedback (minimum 7 days)
3. Maintainers review feedback
4. Decision announced
5. Implementation

**Timeline:** 1-2 weeks

### Major Changes (RFC Process)
For significant changes (governance, architecture, licensing):

1. **RFC Document** created in `docs/rfcs/`
2. **Announcement** in Discussions and README
3. **Community Input** (minimum 30 days)
4. **Maintainer Review** of feedback
5. **Vote** by Maintainers (majority required)
6. **Implementation** if approved

**Timeline:** 1-2 months

## Dispute Resolution

### Data Disputes
When contributors disagree about data accuracy:

1. **Open Issue** with "dispute" label
2. **Present Evidence**: Both parties provide sources
3. **Community Input**: Others can weigh in
4. **Maintainer Decision**: Based on evidence quality
5. **Documentation**: Decision rationale recorded

**Resolution criteria:**
- Primary sources > secondary sources
- Recent data > old data
- Official sources > third-party
- Multiple sources > single source

### Conduct Issues
For behavior violations:

1. **Report** to maintainers (private)
2. **Investigation**: Maintainers review
3. **Warning** or action taken
4. **Appeal** process available

## Transparency

### All Public
- Code changes
- Data changes
- Policy discussions
- Dispute resolutions
- Maintainer decisions

### Private Only
- Security issues
- Personal information
- Conduct reports (until resolved)

## Checks and Balances

### Community Can:
- Fork the project at any time
- Propose removing maintainers
- Challenge any decision
- Appeal dispute resolutions

### Maintainers Cannot:
- Change core charter principles without RFC
- Override community consensus without evidence
- Make decisions in secret
- Benefit financially from their role

### Providers Cannot:
- Pay for better placement
- Edit competitor data
- Override community data without evidence
- Circumvent review process

## Succession Planning

To ensure project longevity:

1. **Multiple Maintainers**: No single point of failure
2. **Clear Process**: For adding new maintainers
3. **Fork-Friendly**: Easy to continue independently
4. **Documented**: All processes written down

## Conflict of Interest

### Maintainers must disclose:
- Employment by providers
- Financial interests in providers
- Personal relationships affecting judgment

### When conflicted:
- Recuse from relevant decisions
- Allow other maintainers to decide
- Document the recusal

## Amendments

This governance document can be amended through:
1. RFC process for major changes
2. Maintainer approval for minor clarifications
3. Community can challenge any change
